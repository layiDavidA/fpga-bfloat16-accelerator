Overview  
The goal of this lab was to design and implement a sequential hardware unit that computes the reciprocal [1/(x)] and the square root [sqrt(x)] of a positive normalized bfloat16 floating-point number using the Newton–Raphson method. The design works by taking a 16-bit bfloat16 number and breaking that into a sign bit, 8-bit biased exponent, and 7-bit fraction fields. The design then uses a seed approximation via ROM, and uses the 5 MSB of the mantissa to index a 32-entry lookup table to fetch an optimized initial numerical guess for the Newton–Raphson method. Next, depending on reciprocal or sqrt, the circuit does two sequential mathematical iterations in Q16.16 format and converges to a true mathematical solution. After the iterations, the exponents are adjusted and shifted depending on whether a reciprocal or a square root was requested. After the mantissa values are finalized and normalization is handled, we convert the final fixed-point results back into 16-bit standard format.

Part I: Reciprocal 
To implement this design, the reciprocal operation was designed and implemented first. Using the lab-provided code in Appendix A, the Python code was used to generate and pre-calculate the initial seeds that the hardware lookup table will use for the reciprocal calculation. Appendix A is needed because the hardware cannot calculate these initial guesses from scratch. In the reciprocal ROM, the 5 MSB of the fractional bits are used to index the ROM. The 5 MSBs give an initial guess that is accurate to about 5 bits of precision and is great for an iterative algorithm like the Newton-Raphson method. 

After the ROM was generated, using the lab-provided code, the top-level skeleton, FSM skeleton, and TB modules were created. The FSM for the reciprocal implementation was made using 8 states. In the first state (IDLE), the design waits for a switch to go high to initiate the reciprocal operation. Next, the design transitions into the second state (UNPACK), where it will slice the 16-bit input (x) into a sign, biased exponent, and fraction bits. In this state, the mantissa is converted to Q16.16 and concatenated with a leading 1 to reconstruct the hidden bit of the normalized floating-point representation, allowing arithmetic operations to be performed on the full significand. In this state, the true mathematical unbiased exponent is also generated by subtracting the biased exponent by 127. Next, the design transitions into the third state (PREP), where it sets the variables needed for the calculations. Next, the design transitions into the fourth state (SEED), where the 5 MSBs of the fractional bits are used to fetch the pre-calculated midpoint reciprocal approximation. After, the design transitions into the states (ITER1) and (ITER2). In ITER1, the design computes the calculation of y1= y0(2-dy0). Next, in ITER2 the design computes calculation y2= y1(2-dy1). These multiplication operations are done using the qmul function provided via the lab document. Next, the design transitions into the seventh state (FINAL), where the exponent and final fixed-point mantissa are handled. Mathematically, in the Newton-Raphson method, the new unbiased exponent gets negated. If the mantissa is not normalized, it is shifted left by 1 bit until it is normalized. To compensate for the shift, the exponent must be subtracted by 1 for each shift. Finally, the mantissa and unbiased exponent get sent to a pack_bfloat16 module. In this module, the exponent is re-biased, the concatenated 1 in the mantissa is dropped, and the mantissa is rounded to 7 bits. Pack_bfloat16 concatenates the sign, exponent, and mantissa to create the final answer to the reciprocal. Finally, the circuit transitions into the state (DONE), and LED[0] is high, and the circuit will await another signal.

To test this design, a testbench was provided via the lab document. The values 1, 1.5, 0.5, 4, and 2 were tested and verified using waveforms for correct functionality. The design was verified and had correct functionality for the reciprocal.

Part II: Square Root 
Next, the square root operation was designed and implemented for this circuit. Using the lab-provided code in Appendix A, the Python code was used to generate and pre-calculate the initial seeds that the hardware lookup table will use for the sqrt calculation. In the sqrt ROM, the line (((in_q - ONE_Q) * 32) / THREE_Q)) was used to index the ROM. This line was used to map the input value from its original range to a valid ROM address range.

The FSM for the reciprocal was updated to allow the sqrt operation. This was done by adding if/else statements to most states. The first state (IDLE) was updated to register a second switch as an input to initiate the sqrt operation. In the second state (UNPACK), the code was kept the same to handle the input x into sign, exponent, and mantissa. For the sqrt operation, in the third state (PREP), if the exponent was negative, the design subtracted by 1 to make it even, and then multiplied the mantissa by two. The odd exponent was checked using an if statement, and checked for the LSB of the exponent bit. Next, the design transitions into the fourth state (SEED), where the sqrt ROM is used to fetch the pre-calculated midpoint approximations. After, the design transitions into the states (ITER1) and (ITER2). In ITER1, the design computes the Newton–Raphson sqrt calculation of y1= (y0/2)(3-cy0^2). Next, in ITER2 the design computes calculation y2= (y1/2)(3-cy1^2). These multiplication operations are done using the qmul function provided via the lab document, and the division by 2 was done by shifting the final expression right by 1. Next, the design transitions into the seventh state (FINAL), where the exponent and final fixed-point mantissa are handled. The exponent gets divided by 2, and the final mantissa is calculated by multiplying the mantissa value from the PREP state by the final sqrt approximation from ITER2 (y2) to reconstruct the true square root. Finally, the mantissa and exponent are sent to the pack_bfloat16 module to create the final answer to the sqrt operation, and the design transitions into the final state (DONE).  

The values 1, 1.5, 0.5, 4, and 2 were tested and verified using waveforms for correct functionality. The design was verified and had correct functionality for the sqrt.

Part III: Physical Implementation
The next step of this design was to implement it on the FPGA board. Using the code from lab 4, a 32x16 RAM and an MIF file were created to hold the decimal values of the input x. In a separate top-level file, the RAM was wired to the top-level skeleton provided in Appendix B. Both the RAM and the circuit were connected via the MAX10 clock. The output of the RAM is wired directly into the input x of the circuit. The RAM address was incremented with KEY[1]. The reset was set to KEY[0], and the sqrt/reciprocal operations were set to SW[1:0]. To prevent bugs in the circuit due to the switches, debouncing was used to eliminate unintended transitions. The RAM address was wired to hex 5, while the 16-bit output was wired to hexes 0-3.

The values 1, 1.5, 0.5, 4, and 2 were tested on the physical board and verified using previous testbench values for correct functionality. The design was verified and had correct functionality for both sqrt and reciprocal.

Part IV: Reciprocal Square Root 
Next, the reciprocal square root operation  [1/sqrt(x)] was implemented on the board. From doing research, it was learned that the sqrt and the reciprocal sqrt were very similar. To implement the reciprocal sqrt, a reg op_rsqrt was created so our design could know when the operation was active. The IDLE state was updated to register SW[0] and SW[1] high as an input to initiate the reciprocal sqrt operation. In the PREP, SEED, and ITER1 state, the only thing that changed was updating the if statements to also watch for op_rsqrt whenever we did anything with the sqrt operation, since they have similar steps. In ITER2, it was made so that if the sqrt was computed, we did exponent/2, but then if the reciprocal sqrt was computed, we did -(exponent/2). In the FINAL state, a separate if statement was put to set the final mantissa value equal to the final value from ITER2. Finally, the mantissa and exponent are sent to the pack_bfloat16 module to create the final answer to our reciprocal sqrt operation, and the design transitions into the final state DONE.  

The values 1, 1.5, 0.5, 4, and 2 were tested and verified using waveforms for correct functionality. The design was verified and had correct functionality for the reciprocal sqrt.

Part V: Timing Optimization
Upon verifying the design on the board, the timing constraints were not met. For 1200mV 85C the design had an Fmax of 25.78 MHz, and for 1200mV 0C the design had an Fmax of 28.25Mhz.  
<img width="623" height="142" alt="Screenshot 2026-07-14 at 9 30 47 PM" src="https://github.com/user-attachments/assets/7bb15838-f893-4852-9af7-59557178ca1c" />





To improve the clock frequency of the design, the critical path was viewed from the Timing Analyzer. It was found that the critical path of our design was from the sqrt ROM to the seed value. We analyzed the sqrt ROM and found that the line (((in_q - ONE_Q) * 32) / THREE_Q)) was slowing down the clock frequency very much. This is because the arithmetic operations involved, specifically subtraction, multiplication, and division, introduce a large amount of combinational delay, increasing the propagation time before the ROM address can be generated and the seed value retrieved. To combat this, we replaced the line with (((in_q - ONE_Q) >> 11 / 3)). This line works better for clock frequency because bit shifting is significantly less expensive than multiplication and division, reducing combinational logic delay in the critical path and allowing the ROM address to be computed faster. As a result, the clock frequency went to 55.1 MHz. Finally, as an attempt to speed up the clock frequency, the iteration states were split into  3 separate states each. This introduced a FF in between each multiplication-heavy state. As a result, this took the clock frequency from 55.1 MHz to 57.3 MHz in 1200mV 85C and 62.34 MHz in 1200mV 0C.

Challenges/Conclusion


In this lab, the hardest part was conceptually understanding the math behind the Newton-Raphson method and also understanding bfloat16 and Q16.16 numbers. This lab successfully demonstrated the implementation of an FPGA-based hardware accelerator to compute operations using the iterative Newton-Raphson method. By partitioning the floating-point components, utilizing a quantized Q16.16 fixed-point datapath, and leveraging seed lookup tables, the multi-cycle FSM design converged rapidly within a minimal cycle count.
